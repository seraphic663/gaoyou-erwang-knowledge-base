# 高邮二王数据库“最小必要版”必要性评估

## 一、结论先行

当前 Demo 采用的 5 表结构：

1. `works`
2. `passages`
3. `terms`
4. `cases`
5. `evidences`

它不是“理论上绝对最少”的方案，但已经非常接近“研究型数据库”的最小必要闭环。

更准确地说：

- 如果只求“先存进去、先展示”，还可以压到 3 表或 4 表
- 如果要体现“文本定位 + 考据过程 + 基础检索 + 证据支撑”，当前 5 表是合理的最小版

## 二、逐表判断

### 1. `works`

- 必要性：高
- 理由：统一管理二王著作、原始经典、工具书等来源
- 若删除：书名、作者、时代信息只能散落在别表文本字段中，后续会很乱

结论：不建议删。

### 2. `passages`

- 必要性：高
- 理由：数据库必须能回到具体文本片段，否则只能存结论，不能做学术追溯
- 若删除：二王原文、原始经典、引证文本会散落在案例叙述里

结论：不建议删。

### 3. `terms`

- 必要性：中
- 理由：有助于把字词对象、术语对象做成稳定检索入口
- 若删除：可暂时并入 `cases`，但术语反查和别名管理会变差

结论：可压缩，但保留更稳。

### 4. `cases`

- 必要性：最高
- 理由：项目核心不是“文本库”，而是“考据库”；一条考据必须作为独立对象存在
- 若删除：问题、方法、过程、结论会分散到别表，数据库会退化成摘录库

结论：不能删。

### 5. `evidences`

- 必要性：中
- 理由：支持一条案例对应多条证据，便于后续反查与权威性分析
- 若删除：可以先塞回 `cases.process_text`，但证据无法结构化管理

结论：可压缩，但保留更符合研究型数据库要求。

## 三、真正的“最小骨架”

如果一定要压到最少，只保留这 3 表也能跑：

1. `works`
2. `passages`
3. `cases`

但那样会带来两个明显问题：

- `terms` 不再能作为独立检索对象
- `evidences` 不再能作为独立证据对象

因此：

- **3 表**：更像极简原型
- **5 表**：更像克制但完整的研究型最小版

## 四、最终判断

当前 5 表里，真正不可删的是：

- `works`
- `passages`
- `cases`

当前 5 表里，真正可以讨论是否先压缩的是：

- `terms`
- `evidences`

一句话概括：

> `cases` 并不多余；如果要继续压缩，优先讨论的应是 `terms` 和 `evidences`，而不是 `cases`。
