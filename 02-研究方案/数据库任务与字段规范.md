# 高邮二王项目数据库任务与字段规范

## 一、任务定义

数据库要解决三件事：

1. 存二王的论述文本
2. 存二王所讨论的原始经典文本
3. 存考据过程、证据和结论，并支持检索

当前最适合的做法不是一步到位做“大而全”数据库，而是先做最小必要版，后续再扩成较完整版。

## 二、最小必要版

适用阶段：现在立刻开工、先做案例录入和检索。

建议 5 张表。

### 1. `works`

用途：统一记录二王著作、原始经典、工具书。

| 字段名 | 说明 |
|---|---|
| `id` | 主键，唯一标识一部书 |
| `title` | 书名，检索展示的主字段 |
| `author` | 作者，如王念孙、王引之、郑玄 |
| `work_type` | 文献类型，建议枚举为“二王著作 / 原始经典 / 工具书 / 研究文献” |
| `dynasty` | 朝代，用于时间筛选与展示 |
| `time_note` | 时间说明，先用文本保存，如“清嘉庆间”“成书于乾隆末年左右” |
| `notes` | 备注，放暂时不值得拆字段的信息 |

### 2. `passages`

用途：记录可以被准确定位的文本片段。  
这一版不拆到字词句段，先以“条目/段落/片段”为单位。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `work_id` | 所属著作，对应 `works.id` |
| `juan` | 卷次；若无卷次可为空 |
| `chapter` | 篇、章、类目等较高层级定位，可为空 |
| `location_note` | 更细的位置说明，如“第12条”“页45行3-5”；这一版先不强行拆字段 |
| `raw_text` | 原始录入文本，尽量贴近底本 |
| `normalized_text` | 规范化文本，用于统一异体、标点、空格等，便于检索 |
| `passage_type` | 片段类型，建议枚举为“二王论述 / 原始经典 / 引证材料” |

### 3. `terms`

用途：记录检索词和方法术语。  
这一版把“字词对象”和“术语对象”放在一张表里，便于先跑起来。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `term` | 词条本体，如“犹豫”“通假”“一声之转” |
| `term_type` | 类型，建议枚举为“字 / 词 / 术语” |
| `category` | 分类标签，如“联绵词”“句法现象”“方法术语” |
| `aliases` | 别名、异体、相关写法；当前阶段可先存文本或 JSON 数组 |
| `notes` | 补充说明，如音韵信息、定义摘要、使用提醒 |

### 4. `cases`

用途：一条记录就是一则完整考据。  
这是最核心的业务表。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `title` | 案例标题，建议写成“问题类型 + 对象”，如“句法与通假：能不我知” |
| `term_id` | 对应主要词条，可为空；有些案例未必只围绕一个词 |
| `erwang_passage_id` | 对应二王论述所在片段，指向 `passages.id` |
| `target_passage_id` | 对应被解释的原始经典片段，指向 `passages.id` |
| `problem` | 问题触发点，即“哪里不通、哪里可疑” |
| `method` | 当前案例的主方法，建议先用文本存，如“改义”“通假”“改形” |
| `process_text` | 考据过程全文说明；最小版先整体存，不拆步骤 |
| `conclusion` | 最终结论，写清“判成什么、怎么解释” |
| `certainty` | 置信状态，建议枚举为“确定 / 可疑 / 待核 / 多解” |
| `status` | 数据状态，建议枚举为“草稿 / 已校对 / 已审核” |

### 5. `evidences`

用途：记录案例所使用的证据。  
一则案例通常对应多条证据，所以应单独成表。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `case_id` | 所属案例，对应 `cases.id` |
| `source_passage_id` | 证据来源文本片段，指向 `passages.id` |
| `evidence_type` | 证据类型，建议枚举为“书证 / 音证 / 义证 / 形证 / 语法证据” |
| `quote_text` | 实际引文；允许和来源片段一致，也允许只截取其中关键部分 |
| `note` | 证据解释，说明这条证据为何有效 |

## 三、最小必要版的关系

最小版的关系很简单：

- `works` 1 对多 `passages`
- `terms` 1 对多 `cases`
- `passages` 通过 `erwang_passage_id` 和 `target_passage_id` 连接 `cases`
- `cases` 1 对多 `evidences`

最小版已经能完成：

1. 按字词检索
2. 看二王原文
3. 看对应经典原文
4. 看考据过程
5. 看证据

## 四、较完整版

适用阶段：案例数量变多，开始区分版本、分层文本、关系网络。

建议 9 张表。

### 1. `works`

保留，用法不变。

### 2. `work_versions`

用途：区分家刻本、点校本、OCR 本等不同版本。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `work_id` | 所属著作，对应 `works.id` |
| `version_name` | 版本名，如“家刻本”“中华书局点校本” |
| `version_type` | 版本类别，建议枚举为“刻本 / 点校本 / OCR本 / 整理本” |
| `editor` | 点校者、整理者、录入来源 |
| `publish_year` | 出版或整理年份 |
| `notes` | 备注，如版本优先级、缺页、异文情况 |

### 3. `text_units`

用途：替代最小版的 `passages`，统一处理字、词、句、段、条目。  
如果进入研究库阶段，这张表会成为基础表。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `version_id` | 所属版本，对应 `work_versions.id` |
| `unit_type` | 单元类型，建议枚举为“字 / 词 / 句 / 段 / 条目” |
| `parent_unit_id` | 上级单元；比如“句”属于某个“段”，“词”属于某个“句” |
| `sequence_no` | 同级顺序，用于恢复原始顺序 |
| `raw_text` | 原文 |
| `normalized_text` | 规范化文本 |
| `location_note` | 定位说明，可与卷、章、页、行一起使用 |

### 4. `terms`

保留，但此时更偏向“规范词条表”。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `term` | 规范词条 |
| `term_type` | “字 / 词 / 术语” |
| `category` | 分类 |
| `notes` | 定义、音韵说明、使用说明等 |

### 5. `term_aliases`

用途：把别名、异体、通写单独管理。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `term_id` | 所属词条，对应 `terms.id` |
| `alias_text` | 别名文本 |
| `alias_type` | 类型，建议枚举为“异体 / 通写 / 别称 / 相关词形” |

### 6. `cases`

保留，但关联对象可以从 `passages` 升级为 `text_units`。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `title` | 案例标题 |
| `term_id` | 主要词条 |
| `erwang_unit_id` | 二王论述所在文本单元 |
| `target_unit_id` | 原始经典所在文本单元 |
| `problem` | 问题触发点 |
| `method` | 主方法 |
| `conclusion` | 结论 |
| `certainty` | 置信状态 |
| `status` | 数据状态 |

### 7. `case_steps`

用途：把过程拆开，支持发疑、取证、释理、结论四步检索。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `case_id` | 所属案例 |
| `step_order` | 步骤顺序，保证过程可恢复 |
| `step_type` | 步骤类型，建议枚举为“发疑 / 取证 / 释理 / 结论” |
| `method_tag` | 这一步对应的方法标签，如“改义”“通假”“双声” |
| `description` | 这一步的具体说明 |

### 8. `evidences`

保留，但证据来源改为指向 `text_units`。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `case_id` | 所属案例 |
| `step_id` | 若这条证据服务于某一步，则关联 `case_steps.id`；否则可为空 |
| `source_unit_id` | 证据来源文本单元 |
| `evidence_type` | 证据类型 |
| `quote_text` | 引文 |
| `note` | 证据说明 |

### 9. `relations`

用途：显式记录通假、同源、一声之转、支持、引证等关系。  
这是从“数据库”走向“知识图谱”的关键表。

| 字段名 | 说明 |
|---|---|
| `id` | 主键 |
| `from_type` | 起点对象类型，如 `term`、`case`、`text_unit` |
| `from_id` | 起点对象 ID |
| `to_type` | 终点对象类型 |
| `to_id` | 终点对象 ID |
| `relation_type` | 关系类型，如“通假”“同源”“一声之转”“支持”“引证” |
| `note` | 对关系的说明 |

## 五、两版怎么选

### 现在建议使用：最小必要版

原因：

1. 表少，录入快
2. 足够支撑网站 Demo 和案例库
3. 不会因为版本、层级、关系设计过重而拖慢进度

### 以后再升级：较完整版

升级信号通常有四个：

1. 开始明显遇到版本问题
2. 开始需要字词句段分层检索
3. 开始需要把过程拆成步骤
4. 开始需要显式关系网络

## 六、最小必要版优先补的字段

如果你们继续沿用当前 demo 六表结构，优先补下面这些字段就够了：

| 表 | 优先补充字段 | 用途 |
|---|---|---|
| `works` | `work_type`, `dynasty`, `time_note` | 解决书的类型和时间信息 |
| `passages` | `chapter`, `location_note`, `passage_type` | 解决逐级定位和文本来源区分 |
| `terms` | `term_type`, `aliases` | 解决词条类型和异名问题 |
| `cases` | `target_passage_id`, `process_text`, `certainty`, `status` | 解决原始经典对应、过程存储和审核状态 |
| `evidences` | `source_passage_id`, `evidence_type`, `note` | 解决证据来源和解释问题 |

## 七、结论

当前最合适的路线是：

1. 先按最小必要版做
2. 先把案例录进去、检索跑通
3. 再根据实际问题升级到较完整版

如果只保留一句最实用的话，就是：

> 现在不要追求表多，而要追求字段够用、关系清楚、能马上录数据。
