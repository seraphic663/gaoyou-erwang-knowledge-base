# 后续工作规划

## 目标

把当前网站从“可展示的 Demo”继续推进到“适合立项、答辩、演示和后续维护的轻量正式版”。

---

## 第一阶段：展示层继续打磨

### 1. 首页继续收口

- 将首页案例区改成更轻的横向精选列表，而不是大卡片矩阵
- 首页首屏增加一句更明确的定位文案：
  - 面向什么研究对象
  - 当前已经完成什么
  - 用户进入后第一步该做什么
- 首页增加“数据库浏览 / 案例库 / 检字流程 / 项目说明”四个主入口卡

### 2. 数据库浏览页进一步专业化

- 将案例浏览改成“档案列表”样式
- 左侧筛选改成更接近数据库的导航方式：
  - 字词分类
  - 案例分类
  - 架构说明
- 右侧加入结果排序：
  - 默认排序
  - 关联案例数排序
  - 证据数排序

### 3. 详情页继续降噪

- 字词详情页增加“重点摘要 / 证据摘录 / 原始字段”三级结构
- 案例详情页增加“结论优先模式”，把过程文本默认折叠
- 证据列表支持“只看某一类证据”的局部筛选

---

## 第二阶段：检索能力升级

### 1. 搜索交互升级

- 首页搜索增加模式切换：
  - 搜字词
  - 搜案例
  - 全文检索
- 数据库页搜索支持即时提示和空态建议
- 搜索结果为空时，给出推荐关键词或邻近分类入口

### 2. 检索维度升级

- 增加按类别筛选与关键词组合检索
- 增加按证据类型筛选
- 增加按案例状态筛选
- 增加按卷册、类目、文献来源筛选

### 3. 检索结果体验升级

- 结果中高亮命中片段
- 结果支持分页或“继续加载”
- 支持从结果直接跳到对应详情页中的命中位置

---

## 第三阶段：数据表达升级

### 1. 数据结构继续规范

- 逐步减少 JSON 字段在业务层的直接暴露
- 补充更清晰的浏览专用 payload，避免前端自己拼装逻辑
- 为后续分页、排序、筛选预留标准参数结构

### 2. 数据质量层支持

- 增加“草稿 / 已审 / 已核”状态展示
- 增加证据来源说明字段的可视化
- 增加缺失字段提醒，例如：
  - 无原始片段
  - 无目标片段
  - 无过程文本

### 3. 研究表达支持

- 字词详情页加入“相关字词”区
- 案例详情页加入“相关案例”区
- 增加一个轻量关系视图，先做“同类案例推荐”，不必一开始就上知识图谱

---

## 第四阶段：项目汇报友好化

### 1. 答辩页面支持

- 首页增加“项目成果速览”模块
- 新增一页“项目亮点”页面，适合老师快速浏览
- 新增“数据规模 / 结构 / 功能 / 演示路径”四段式说明页

### 2. 文档配套

- README 拆成：
  - 启动说明
  - 部署说明
  - 数据源说明
  - 页面结构说明
- 增加“网站迭代记录”和“后续工作规划”的互链

### 3. 演示路径设计

- 设计 3 分钟演示路径
- 设计 5 分钟答辩路径
- 设计“首页 -> 数据库浏览 -> 案例详情 -> 返回首页”的标准展示流程

---

## 第五阶段：工程层继续完善

### 1. 前端工程

- 将首页脚本、数据库浏览脚本、详情页脚本再进一步模块化
- 提取公共渲染函数，减少重复 HTML 拼接
- 增加更稳定的页面初始化和错误提示机制

### 2. 后端接口

- 为浏览接口加入分页参数
- 为搜索接口加入模式参数统一定义
- 为详情接口加入更多统计字段

### 3. 部署与维护

- 补充 GitHub 提交规范
- 补充 Railway 更新流程
- 明确“更新 SQLite -> 导出快照 -> 推送部署”的操作文档

---

## 建议优先级

### P1：马上做

- 案例浏览页改成更轻的档案列表
- 数据库页增加分页或继续加载
- 搜索空态、加载态完善

### P2：下一轮做

- 首页成果速览模块
- 结果命中高亮
- 相关案例 / 相关字词推荐

### P3：中后期做

- 关系视图
- 更完整的研究工具页
- 更正式的管理与审核工作流

---

## 推荐的下一步

如果只继续做一轮，我建议优先做下面这一组：

1. 案例浏览页改成“档案列表”
2. 数据库页加分页
3. 搜索结果高亮
4. 首页增加“项目成果速览”

这一组最能直接提升观感、演示质量和专业度，同时工作量仍然可控。
