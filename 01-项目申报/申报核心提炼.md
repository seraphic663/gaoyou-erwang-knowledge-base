# 申报核心提炼

## 一、这份文件的用途

本文件用于整合原 `feedback.md`、`申报文本示范.md`、`写作指南.md` 三份材料中的有效内容，作为 `01-项目申报/立项申报.md` 的辅助母稿。

整合原则：

1. 保留对正式申报最有帮助的结构、论点和判断
2. 删除重复展开、过细示例、可后续再补的长篇模板
3. 单独保留评审视角下的风险提醒，避免申报文本只顾“写满”而忽略“可评”

## 二、项目申报的核心论证线

整份申报最重要的逻辑主线应当稳定为：

**学术对象重要** → **现有工具有缺口** → **二王方法可结构化** → **项目能落地实现** → **成果具有学术与平台双重价值**

可以进一步压缩为一句话：

> 本项目并非一般意义上的古籍全文检索平台，而是以高邮二王为对象，尝试将其考据过程、证据链条与方法规则结构化表达，从而把传统个体化的学术经验转化为可检索、可复盘、可积累的数字人文学术资源。

## 三、正式申报中必须保留的核心内容

### 1. 项目简介部分必须说清的四点

项目简介最重要的不是写全，而是写准。必须包括：

1. 二王及“王氏四种”的学术地位
2. “形、音、义知一推三”“经字互证”“一声之转”的方法核心
3. 现有古籍平台止于全文检索，不能呈现推理过程
4. 本项目要把“结论”升级为“过程+规则+证据链”的结构化平台

简介部分推荐语气：

- 先讲二王的重要性
- 再讲数字化工具的缺口
- 最后讲本项目的独特切入点

### 2. 项目申请条件部分必须说清的三点

这一部分不是列简历，而是证明“团队和项目是匹配的”。

建议按能力而非按姓名展开：

1. 学术基础：古文字、音韵、训诂、古籍阅读能力
2. 技术能力：检索、数据库、数据处理、可视化能力
3. 交叉能力：有人能连接学术层与工程层

写法原则：

- 不要平均用力
- 要突出“谁负责什么”
- 要把已有前期工作写成可验证的积累

### 3. 立项依据与文献综述部分必须说清的三层结构

这一部分建议固定成三层：

1. 宏观层：古籍数字化和数字人文的背景
2. 中观层：二王研究与古籍知识图谱已有成果
3. 微观层：现有研究的缺口恰好落在“方法过程尚未结构化”

必须强调的关键区别是：

- 现有平台主要处理“对象知识”
- 本项目试图处理“方法知识”

这是整份申报最有辨识度的创新表述。

### 4. 研究方案部分必须说清的四个阶段

最稳妥的结构就是：

1. 阶段 A：数据准备
2. 阶段 B：基础检索
3. 阶段 C：证据关系与案例标注
4. 阶段 D：规则提炼与类推探索

其中当前最该强调的是 A-C，D 可以写成“探索性阶段”或“可选深化阶段”，不要写成必须完成的刚性成果。

### 5. 项目创新部分建议固定为三个维度

最稳的写法是：

1. 从“结果存储”到“过程追踪”
2. 从“孤例处理”到“规则类推”
3. 从“个人积累”到“协作共建”

这三条已经足够支撑创新性表述，不必再无限扩展。

## 四、最值得保留的申报表达

以下几类表达应当保留，因为它们最能体现项目特色：

### 1. 对问题的定义

当前古籍数字化平台解决的主要是“某字在哪里出现”的问题，而未能解决“学者如何通过多步推理将零散证据串联为解释结论”的问题。

### 2. 对项目创新的定义

本项目不是单纯建设“古籍数据库”，而是尝试把二王的考据过程、证据链与判断规则显式化，使之成为可检索、可复盘、可迁移的结构化学术资源。

### 3. 对平台目标的定义

平台的目标不是替代学者，而是辅助学者：通过结构化标注与规则提炼，帮助研究者更快发现问题、追踪证据、比较同类案例。

## 五、评审最看重的内容

从 `feedback.md` 的意见看，评审最看重的其实不是“你写得多漂亮”，而是以下四点：

1. 研究问题是否清楚
2. 技术路线是否可行
3. 前期基础是否可信
4. 承诺是否克制、指标是否有依据

可以把它们理解成答辩时的四个核心判断轴。

## 六、现阶段最可能被质疑的硬伤

这是从反馈中提炼出来、最值得提前修补的部分。

### 1. 文献综述不够硬

风险：

- 研究现状写得像“知道一些背景”
- 但没有真正证明“现有工作为什么不能解决本项目的问题”

应对：

- 用更明确的对比写法
- 不只说“已有平台存在不足”，而要说清“不足具体在哪里”

### 2. 若干量化指标依据不足

风险：

- 例如准确率、效率提升倍数、并发量、推荐精度等数字，如果没有测量口径，会显得虚

应对：

- 能验证的再写具体数字
- 暂时不能验证的，改成“初步测试”“计划评估”“实现后评测”

### 3. 后期目标写得偏满

风险：

- 如果一开始就把智能推荐、规则迁移、开放共建全部写成刚性交付，评审容易觉得超前

应对：

- 把“平台基础功能”和“后续扩展目标”分开
- 让 A-C 成为必须完成，D 成为探索性成果

### 4. 预算中过于边缘的支出容易被质疑

风险：

- 如果出现与核心研究关联较弱的设备或工具，会削弱说服力

应对：

- 经费集中在文献、数据库、算力、咨询、资料整理这些核心项
- 其余支出尽量克制

## 七、最推荐的写作原则

### 1. 具体大于抽象

不要只说“本项目很有创新性”，而要具体说明创新体现在“过程追踪”“规则提炼”“案例共建”。

### 2. 可行大于宏大

不要把项目写成“虚拟学者系统”，而应写成“古籍研究辅助平台”。

### 3. 证据大于口号

凡是涉及前期基础、实验结果、团队能力，尽量给出可验证的案例、数量和经历。

### 4. 阶段清晰大于一次到位

申报书最容易出问题的地方，不是“目标太低”，而是“把所有远景都写成当前任务”。

## 八、可以直接指导后续修改的清单

后续继续改 `立项申报.md` 时，建议按这张清单检查：

1. 是否明确区分了“对象知识”和“方法知识”
2. 是否明确写出了现有平台的能力边界
3. 是否把研究方案拆成了清楚的阶段
4. 是否避免了无法证明的夸张指标
5. 是否把“自动判断”改写为“辅助推荐 + 专家复核”
6. 是否把创新点收束为 3 条左右，而不是铺得太散
7. 是否让前期基础、团队分工、预算和成果之间互相对应

## 九、一句话结论

如果只保留一句最重要的话，这三份文件合起来真正要告诉你们的是：

> 这份申报最有竞争力的地方，不是把项目写成“大而全的 AI 平台”，而是把它写成一个以高邮二王为对象、以考据过程结构化为核心创新、并且在一年周期内能够分阶段落地的数字人文学术项目。
