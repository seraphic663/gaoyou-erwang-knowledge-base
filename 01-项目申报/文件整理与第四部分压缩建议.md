## 二、第四部分超字数分析

### 1. 当前判断

目标文件是：
[立项申报v4.2（格式修正）.docx](D:/26大创/01-项目申报/v4.2/立项申报v4.2（格式修正）.docx)

第四部分范围是：
`四、项目研究方案` 到 `五、项目特色与创新之处` 之前。

按当前 `docx` 逐段抽取后统计：

| 统计口径 | 估算字数 |
|---|---:|
| 含“四、项目研究方案”标题与说明行 | 4436 |
| 不含主标题 | 4428 |
| 不含主标题与说明行 | 4356 |

结论：

- 不管按哪种口径算，第四部分都超了 `4000` 字。
- 我这里实测是约 `4356-4436` 字，不是你说的 `4800`。两者不冲突，通常是因为 Word 的字数统计口径、是否把标题/说明/标点/数字算入，会有差异。
- 从压缩策略看，不需要“伤筋动骨”重写；只要压掉 `400-600` 字，第四部分就能稳稳落进限制线。

### 2. 最适合压缩的地方

下面按“优先删哪里最不伤主线”排序。

| 优先级 | 位置 | 当前问题 | 建议动作 | 预计可减 |
|---|---|---|---|---:|
| 高 | `3. 研究内容` 中两段图片占位 | 这两段更像答辩提示，不是申报书核心研究方案内容 | 直接移出正文，改存到单独说明文件或修改报告 | 320-360 |

| 中 | `4. 研究方法` 总述段 | 文献来源交代稍满，与文综部分有回声 | 压成“文献细读 + 结构拆解 + 数据建模，方法依据王宁与考据结构研究” | 50-80 |

| 低 | `2. 研究目标` 首段 | 可略压，但不是主要肥肉 | 可微调，不建议先动 | 20-40 |

### 3. 最建议先删的具体段落

#### 方案 A：最稳妥压缩

如果目标只是“快速回到 4000 字以内”，最建议先动这三处：

1. 删除第 66-68 段图片占位说明。
2. 压缩第 121 段技术可行性长段。
3. 压缩第 63、65 段中关于演示站和首页样本的解释句。

这样通常就能减掉约 `500-700` 字，而且不会伤到项目核心论证。

#### 方案 B：尽量保留展示感

如果你想保留“网站演示已经存在”的说服力，那就：

1. 不删“造舟于河”这个样本。
2. 删除图片占位 2 段。
3. 把技术可行性中的细数据全部压成一句。
4. 把“知识库与演示系统实现”改成更硬的摘要句。

这样仍能保留“项目已经可演示”的气质，但文字不会像修改日志。

### 4. 不建议优先动的部分

| 位置 | 原因 |
|---|---|
| `1. 研究问题` 的四个子问题 | 这是第四部分的骨架，删多了会让后文失去抓手 |
| `2. 研究目标` 的五个目标条目 | 这是申报书最容易被评审快速把握的部分，宜稳住 |
| `6. 可行性分析` 第一、三、四段 | 这三段在“学术可行性 / 组织可行性 / 时间可行性”上分工清楚，压缩收益不如技术段大 |
| `7. 风险与应对` | 这块字数不大，但态度很重要，删了反而显得准备不足 |

### 5. 可直接参考的删改顺序

建议实际修改时按这个顺序动：

1. 先拿掉图片占位与图下说明。
2. 再压“技术可行性较强”长段。
3. 再压“知识库与演示系统实现”与“案例验证与前端化表达”。
4. 若还超，再微调“研究方法”总述。

这样最稳。

## 七、冗余类型补充判断

上一版主要写了“目录归类冗余”，但还不够。更完整的判断，应至少分成四类。

### 1. 结构冗余

| 对象 | 问题 | 结论 |
|---|---|---|
| `imgs/imgs/` | 双层同名目录，没有信息增量 | 这是最典型的结构冗余，应展平 |
| 根目录同时放主稿、依据文件、过程报告 | 层级职责不清 | 属于结构冗余，应分层 |
| 根目录同时放 `立项申报v4.2.docx` 与 `立项申报v4.2（格式修正）.docx` | 两个都像“当前版”，会造成判断歧义 | 属于结构冗余，应明确主副关系 |

### 2. 流程冗余

| 对象 | 问题 | 结论 |
|---|---|---|
| `归档/build_v42_docx.py` | 该脚本当前位于 `归档/`，但脚本内部仍把 `BASE_DIR` 当作源文件目录，并假定 `立项申报v4.2.md`、模板 `docx`、输出文件都与它同级 | 这不是普通“归档文件”，而是一个已错位的过程工具；若不调整路径或移动回正确位置，后续容易误用 |
| `立项申报v4.2.docx` 与 `立项申报v4.2（格式修正）.docx` | 一个是脚本生成版，一个是 Word 格式修正版；若不说明谁是最终源，后续可能来回覆盖 | 属于流程冗余，应规定唯一主版本 |

### 3. 版本冗余

| 对象 | 问题 | 结论 |
|---|---|---|
| `归档/立项申报v4.1（格式调整版）-批注落实副本.docx` 与带“副本 - 副本”的同名文件 | 已确认完全相同 | 明确重复，应删其一 |
| `归档/立项申报v4.1.md`、`归档/立项申报v4.1（格式调整版）.docx`、`归档/立项申报v4.1（格式调整版）-批注落实副本.docx`、`立项申报v4.2.docx`、`立项申报v4.2（格式修正）.docx` | 这些不是重复垃圾，而是版本链；但现在平放后看起来像并列可用文件 | 属于版本冗余中的“未编目冗余”，应改成可读版本链 |

### 4. 语义冗余

| 文件组 | 问题 | 结论 |
|---|---|---|
| `要求.md` + `过程性知识呈现与训诂考据整理.md` | 都围绕“过程性知识”展开，前者像短任务说明，后者像完整展开稿 | 信息高度相关，可并包管理 |
| `注意事项.md` 与上述两份文件 | 都在指导 `v4.2` 如何写，只是粒度不同 | 不适合继续散放，应作为“写作依据组”整体管理 |
| `归档/考据知识.md` + `归档/考据结构与具体实例.md` + `归档/训诂术语.md` + 两个对应 `docx` | 都在提供概念、结构、术语层的支撑材料，只是拆散了 | 适合整合为“概念资料组” |

### 5. 哪些不是冗余

也需要反过来说明，避免误删：

| 对象 | 为什么不应简单视作冗余 |
|---|---|
| `立项申报v4.2.md` 与 `立项申报v4.2（格式修正）.docx` | 它们承担的是“内容底稿”和“正式版式稿”两种职责，关键不是删，而是指定主从关系 |
| `prewrite_revision_log.md` 与 `prewrite_writing_preferences.md` | 一个记“落实了什么”，一个记“推断出什么偏好”，功能不同 |
| `考据结构.jpg` 与网站最新架构图 | 虽然都叫“结构图”，但前者更像早期概念图，后者是网站现行释读架构图，不宜未经比对就删 |

## 十、结论

### 1. 关于第四部分

第四部分确实超字数，但并没有超到必须重写的程度。最该动的不是“研究问题”和“研究目标”，而是：

1. 图片占位说明
2. 技术可行性长段
3. 研究内容中偏展示化、记录化的说明句

只要按这个顺序压，落回 `4000` 内是完全可做的。

### 2. 关于文件夹整理

当前目录的问题不只是“文件太多”，而是四类冗余同时存在：

1. 结构冗余
2. 流程冗余
3. 版本冗余
4. 语义冗余

因此最优解不是乱删，而是：

1. 根目录只留当前主稿
2. 参考材料单独成组
3. 旧稿与过程文件留归档
4. 图片目录展平
5. 对内容明显重合的文件先整合，再决定删不删
6. 仅删除已证实的重复副本
