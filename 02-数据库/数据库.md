# 高邮二王项目数据库说明

这份文档按仓库当前真实实现重写，不再使用早期“方案稿”口径。

它描述的是现在实际在跑的数据库链路、当前五表结构、网站读取方式，以及与更理想研究库之间的边界。

## 一、当前真实目录

数据库相关内容现已统一放在 [02-数据库](D:/26大创/02-数据库)：

- `数据库.md`：当前数据库说明
- `source.txt`：原始语料
- `parser2.py`：文本解析脚本
- `parsed_full.py`：解析结果
- `bulk_importer.py`：批量导入脚本
- `db2.py`：SQLite 五表 schema 与检索接口
- `data2/dictionary.db`：当前主用 SQLite 实库

网站在 [03-项目网站](D:/26大创/03-项目网站)，默认读取：

1. `03-项目网站/data/sqlite-snapshot.json`
2. 如果快照不可用，再回退到 `03-项目网站/data/demo-db.json`

快照由下面这条链路生成：

`02-数据库/data2/dictionary.db` -> `python 03-项目网站/scripts/sqlite_bridge.py export` -> `03-项目网站/data/sqlite-snapshot.json`

## 二、当前不是“数据库直连网站”，而是“SQLite -> 快照 -> 网站”

这点必须说清。

当前网站虽然以 SQLite 实库为真实数据源，但 Node 服务并不直接查询 SQLite。当前真实架构是：

1. SQLite 实库保存在 `02-数据库/data2/dictionary.db`
2. `03-项目网站/scripts/sqlite_bridge.py` 读取这份 SQLite
3. 脚本导出统一 JSON 快照 `03-项目网站/data/sqlite-snapshot.json`
4. 网站后端优先读取这份快照
5. 前端再通过网站 API 读取统一结构

所以当前项目的“数据库架构”，实际上分两层：

- 存储层：SQLite 五表实库
- 消费层：网站读取统一 JSON 快照

## 三、当前实际采用的是五表结构

当前真实 schema 以 [db2.py](D:/26大创/02-数据库/db2.py) 和网站侧 [store-definitions.js](D:/26大创/03-项目网站/src/store-definitions.js) 为准。

五张主表是：

1. `works`
2. `passages`
3. `terms`
4. `cases`
5. `evidences`

网站 API 也是围绕这五张表组织的。

## 五、当前真实关系

现在的关系不是纯教科书式三范式，而是“研究可用 + 网站可展示”的折中实现。

主要关系如下：

- `works` 1 对多 `passages`
- `passages` 通过 `erwang_passage_id` 和 `target_passage_id` 连接 `cases`
- `cases` 1 对多 `evidences`
- `terms` 与 `cases` 当前不是独立关联表，而是通过 `cases.term_ids` 和 `terms.case_ids` 两个 JSON 字段互相关联
- `evidences` 还直接通过 `term_id` 与 `terms` 关联

这意味着当前实现有两个特点：

1. 没有单独的 `case_terms` 中间表
2. 某些多对多关系是用 JSON 字段保存的

这不是最规范的研究数据库设计，但对当前网站原型是可接受的。

## 六、当前实现里额外存在的搜索层

除了五张主表，当前 SQLite 里还建了三张 FTS 虚拟表：

- `terms_fts`
- `passages_fts`
- `evidences_fts`

它们的作用是提供全文检索支撑，不属于网站主 schema 展示层，但属于当前真实数据库实现的一部分。

## 七、当前数据状态

按当前实库和快照的状态，网站已经可以稳定读取：

- `works`：已有数据
- `terms`：已有大量数据
- `cases`：已有大量数据
- `evidences`：已有大量数据
- `passages`：结构已存在，但当前仍基本为空

这说明当前项目的现实重点是：

- 词条
- 案例
- 证据

而不是完整的“原文片段层”。

换句话说，当前库已经足够支撑网站展示和检索，但还没有走到“文本全层级建模完整”的阶段。

## 八、当前网站如何使用这套数据库

当前网站并不暴露 SQLite 细节给前端，而是统一输出五表语义。

网站目前支持的主要能力有：

1. 返回数据库统计信息
2. 返回五表 schema 说明
3. 按关键词检索词条和案例
4. 查看单个词条详情
5. 查看单个案例详情
6. 浏览数据库内容

所以现在这套数据库的主要目标不是学术上“建到最完整”，而是：

- 数据关系能讲清
- 词条和案例能查
- 证据链能展示
- 申报书与答辩时能说明“数据库 - 接口 - 页面”的闭环

## 九、当前版本与早期方案的主要差异

为了避免以后文档继续写偏，这里把最关键的偏差直接列出来。

### 1. 不是单个 `term_id`，而是 `term_ids`

早期方案里 `cases` 常按单词条建模，但当前真实实现已经是多词条关联。

### 2. 不是纯规范化设计

当前实现为了网站展示效率，保留了：

- `terms.case_ids`
- `cases.term_ids`
- `aliases` JSON
- `core_meaning`
- `core_snippet`

所以它更像“可运行原型库”，而不是最终研究级标准化库。

### 3. `passages` 已建表，但尚未真正成为核心数据层

当前 schema 里有 `passages`，但目前并没有把大量案例都还原成完整的片段网络。

### 4. 网站当前读取的是快照，不是实时 SQLite 查询

这一点在结构说明中必须一直保持准确。

## 十、如果以后升级，应该往哪里升

当前版本已经够支撑原型网站和申报展示。以后如果继续升级，优先顺序建议是：

1. 先把 `passages` 补实，让案例、原文、证据来源形成稳定片段层
2. 再把 `terms` 与 `cases` 的 JSON 关联改成独立关联表
3. 之后再考虑版本层、关系层、知识图谱层

也就是说，下一步不是盲目加表，而是先把现在已有五表真正填实。

## 十一、一句话结论

当前项目真实使用的是一套“SQLite 五表实库 + JSON 快照网站消费层”的轻量架构。它已经能稳定支撑词条、案例和证据展示，但仍属于研究原型阶段，不应再用早期概念稿去描述它，而应以当前代码和实库字段为准。
